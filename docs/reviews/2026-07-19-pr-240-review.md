## F6 (P2, @blorbeer-cmd) – Mutationen nicht auf den aktuellen Event-Scope begrenzt

- Befund bestätigt: `resolveChecklistItem`/`resolveChecklistTask` (beide `resolveGroupResource`)
  prüfen ausschließlich Gruppenmitgliedschaft, nie `event_id`. Nach einem Eventwechsel
  innerhalb derselben Gruppe blieb eine alte, bekannte ID weiterhin mutierbar (Position
  abhaken/löschen, Aufgabe claimen/erledigen/zurückziehen), obwohl `GET` sie im aktuellen
  Scope nicht mehr auflistet.
- Fix: neue Hilfsfunktion `currentEventScope(req, res)` in `checklist.ts`, aufgerufen direkt
  nach dem Laden der Ressource in allen sechs ID-basierten Mutationsrouten
  (`PATCH`/`DELETE /items/:id`, `claim`/`release`/`done`/`DELETE /tasks/:id`); bei
  `resource.event_id !== scope.eventId` `404` (dieselbe Meldung wie "nicht gefunden", da eine
  veraltete ID keinen anderen legitimen Zweck hat).
- Test: `api.groupChecklist.required.test.ts` erweitert um ein Zwei-Event-Szenario in
  *derselben* Gruppe (Event A → Event A2 wechseln), das genau diese Lücke reproduziert
  (Mutation auf `itemA`/`taskA` nach dem Wechsel → `404`) und bestätigt, dass ein frisch unter
  Event A2 angelegtes Item weiterhin normal mutierbar bleibt.

## F7 (P2, Codex) – Checklist-Caches nicht bei Eventwechsel invalidiert

- Kommentar: Zeile 71, `server/public/js/views/checklist.js`
- Befund bestätigt: `app.js`s `events:changed`-Handler lädt Basisdaten neu und invalidiert
  mehrere andere View-Caches, aber nicht `checklist.js`s eigene `tasksCache`/`itemsCache`.
  Start/Stop/Wechsel des getrackten Events ließ die vorherigen Zeilen sichtbar und anklickbar.
- Fix: `invalidateChecklist()` in `app.js`s `fullReloadEvents`-Handler ergänzt (derselbe Ort,
  an dem bereits `invalidateHomeSeating`/`invalidateSeating`/etc. für dieselben Events laufen).

## F8 (P2, Codex) – Zuweisungs-Auswahl nicht auf die aktuelle Gruppe begrenzt

- Kommentar: Zeile 197, `server/public/js/views/checklist.js`
- Befund bestätigt: Das "Aufgabe verteilen"-Modal baute seine Checkbox-Liste aus
  `state.players` (globales, gruppenübergreifendes Roster), während der Server
  `assigneePlayerIds` über `activeGroupPlayers(req.group!.id, …)` validiert. In
  Multi-Group-Required-Setups konnte ein Organisator so eine sichtbare, aber gruppenfremde
  Person auswählen, woraufhin der gesamte Request `404`e.
- Fix: neue Funktion `assigneeCandidates()` lädt `api.groups.members(groupId)` (liefert
  bereits `{playerId, name, color, avatar}` der aktiven Gruppenmitglieder) und fällt auf
  `state.players` zurück, wenn kein `groupId` in `sessionStorage` steht oder der Aufruf
  fehlschlägt – das deckt Legacy-Modus ab, wo `/api/groups/:id/members` mangels Session
  grundsätzlich `404`t (siehe `requireGroupMembership`) und es ohnehin nur die eine implizite
  Gruppe gibt.
- Test: manuell im Browser (Legacy-Modus, siehe Verifikation unten) – die Auswahlliste öffnet
  weiterhin normal und fällt korrekt auf das globale Roster zurück. Kein automatisierter
  Required-Mode-Test ergänzt, da das Modal selbst (im Gegensatz zu den Server-Routen) keine
  Sicherheitsgrenze ist, sondern nur die Auswahl komfortabler macht – der Server validiert
  `assigneePlayerIds` bereits unabhängig davon.

## F9 (P2, Codex) – Push-Topic beim Freigeben nicht aufgelöst

- Kommentar: Zeile 514, `server/src/routes/checklist.ts`
- Befund bestätigt: Der `release`-Handler aktualisierte den Task-Status, rief aber nie
  `resolvePushTopic` auf. Das "dir wurde zugewiesen"-Banner der freigebenden Person blieb im
  Benachrichtigungscenter aktiv, bis (falls überhaupt) jemand anders den Task claimt/erledigt/
  zurückzieht.
- Fix: `resolvePushTopic('checklist-task:' + task.id, false, …)` direkt nach dem
  `UPDATE ... SET status = 'open'` ergänzt, analog zu `claim`/`done`/`cancel`.

## F10 (P2, Codex) – Packliste fehlte in den Feed-Deep-Links

- Kommentar: Zeile 336, `server/src/routes/checklist.ts`
- Befund bestätigt: `pushFeed.js`s `FEED_LINK_LABELS` kannte `checklist` nicht, obwohl
  `checklist.ts` bereits `/#checklist`-Links in seinen Push-Payloads verschickt. Gespeicherte
  Einträge im Benachrichtigungscenter/Kiosk-Banner zeigten das generische Glocken-Icon ohne
  "Zur Packliste"-Aktion.
- Fix: `checklist: 'Zur Packliste'` in `FEED_LINK_LABELS` ergänzt; `domainIcons.js` hatte den
  passenden `checklist: 'clipboard'`-Eintrag bereits aus PR #237.
